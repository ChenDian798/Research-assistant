# 创新点查新后端优化交接方案

## 1. 背景与目标

当前项目已经具备第一版“创新点查新”闭环：

- 前端入口：创新点查新页面
- 后端接口：`POST /api/novelty-check`、`GET /api/novelty-check/{job_id}`
- 后端评估：`src/research_agent/novelty_check.py`
- 复用能力：普通文献检索、筛选、去重、历史记录、异步任务、前端轮询

但第一版仍然把“查新检索”当成“普通文献检索”来做，导致检索式过窄、候选池为空、用户看到“证据不足”。本方案目标是把查新功能升级为专用后端流程：

1. LLM 先做创新点结构化拆解，但不直接生成唯一检索式。
2. 后端生成多层检索计划，包括宽检索、核心主题检索、方法检索、场景检索、baseline 检索。
3. 每个来源使用更适合自身的查询策略，尤其 PubMed 不使用过度 AND。
4. 候选池保留分层结果，不只保留普通检索的 qualified/needs_review。
5. 查新报告显示“检索诊断”，让用户知道是没检索到、源失败、还是被过滤。
6. 最终输出不是“是否创新”的绝对判断，而是“重合风险 + 证据 + 需人工核验点”。

## 2. 当前问题诊断

### 2.1 架构问题

当前查新任务在 `web_app.py` 中的流程是：

```text
_handle_novelty_check
  -> _run_novelty_check_job
    -> _run_literature_search_pipeline
      -> search_papers
      -> screen_references
      -> apply_relevance_gate
      -> verify_references
    -> NoveltyCheckWorkflow.run(references=qualified + needs_review)
```

问题是：`_run_literature_search_pipeline` 是为“找正式候选文献”设计的，不是为“查新穷举相似工作”设计的。它会较早过滤掉 off-topic 或低相关候选，而查新其实需要保留“可能相邻但低置信”的候选，交给查新评估器判断。

### 2.2 检索式问题

普通文献检索的 LLM query planner 目前输出单个主检索式和一个 PubMed 检索式。查新场景下，如果把所有创新元素都 AND 在一起，会非常容易返回 0。

失败示例：

```text
ischemic stroke
AND non-contrast CT
AND lesion segmentation
AND lightweight/efficient
AND domain generalization/cross-domain/few-shot/small sample
```

真实相关论文可能只满足其中三四个条件，例如“stroke CT segmentation”，但不会同时在标题/摘要中写 lightweight、domain generalization、few-shot。

### 2.3 输入污染问题

用户可能把“验收重点”“备注”“导师要求”“测试说明”也粘进输入框。当前 `NoveltyCheckWorkflow.extract_claims()` 会把这些也拆成创新主张，进而污染检索和报告。

### 2.4 可观测性问题

前端报告目前只显示“候选总数 0”，没有清楚展示：

- 哪些来源成功/失败
- 每个来源返回多少条
- 多少条被过滤
- 为什么没有进入查新对照
- 是否检索式过窄

这会让用户误以为“系统没输出”。

## 3. 后端目标架构

建议新增一个查新专用检索规划模块：

```text
src/research_agent/novelty_planner.py
```

建议新增一个查新专用检索执行模块：

```text
src/research_agent/novelty_search.py
```

最终后端流程改成：

```text
POST /api/novelty-check
  -> normalize user input
  -> NoveltyPlanner.extract_plan()
  -> NoveltySearchRunner.run_plan()
  -> candidate pool merge/dedupe
  -> light screening, not hard rejection
  -> NoveltyCheckWorkflow.run()
  -> attach search diagnostics
  -> save history/job result
```

不要继续让查新入口直接调用 `_run_literature_search_pipeline()` 作为唯一候选来源。普通检索链路可以复用底层 `search_papers` 的 API 适配器、normalize、dedupe 工具，但不能复用严格筛选结果作为唯一输入。

## 4. 核心数据结构

### 4.1 InnovationPlan

LLM 不直接返回单个检索式，而返回结构化计划：

```json
{
  "language": "zh",
  "clean_innovation_text": "用户真正的创新点正文，去掉验收重点/备注",
  "ignored_context": ["验收重点", "导师要求", "测试说明"],
  "domain": "biomedical|computer|engineering|society",
  "core_claim": "低资源急性缺血性卒中 NCCT 病灶分割",
  "required_concepts": [
    {"term": "ischemic stroke", "type": "condition", "must_keep": true},
    {"term": "non-contrast CT", "type": "modality", "must_keep": true},
    {"term": "lesion segmentation", "type": "task", "must_keep": true}
  ],
  "method_concepts": [
    {"term": "lightweight encoder", "type": "method"},
    {"term": "cross-domain consistency", "type": "method"}
  ],
  "context_concepts": [
    {"term": "low-resource", "type": "data_setting"},
    {"term": "small hospital dataset", "type": "data_setting"}
  ],
  "baseline_concepts": ["nnU-Net", "UNet++", "SwinUNETR"],
  "forbidden_noise": ["验收重点", "报告应提示"]
}
```

### 4.2 NoveltySearchQuery

后端基于 InnovationPlan 生成多组查询：

```json
{
  "query_id": "pubmed_broad_topic",
  "purpose": "broad_topic|core_topic|method_overlap|context_overlap|baseline_overlap",
  "sources": ["pubmed"],
  "query": "(ischemic stroke[Title/Abstract] OR cerebral infarction[Title/Abstract]) AND (CT[Title/Abstract] OR computed tomography[Title/Abstract] OR NCCT[Title/Abstract]) AND (segmentation[Title/Abstract])",
  "strictness": "broad|medium|narrow",
  "max_results_per_source": 10
}
```

### 4.3 NoveltyCandidate

候选必须保留来源和命中原因：

```json
{
  "title": "...",
  "abstract": "...",
  "source": "...",
  "doi": "...",
  "retrieved_from": "pubmed",
  "matched_query_ids": ["pubmed_broad_topic", "semantic_method_overlap"],
  "retrieval_purpose": ["broad_topic", "method_overlap"],
  "candidate_status": "strong_candidate|weak_candidate|source_noise",
  "screening_notes": ["missing method concept", "matched core topic only"]
}
```

### 4.4 SearchDiagnostics

报告必须带诊断信息：

```json
{
  "queries": [
    {
      "query_id": "pubmed_broad_topic",
      "source": "pubmed",
      "query": "...",
      "returned": 8,
      "kept": 5,
      "filtered": 3,
      "error": ""
    }
  ],
  "source_summary": {
    "pubmed": {"returned": 8, "kept": 5, "error": ""},
    "semantic": {"returned": 0, "kept": 0, "error": "HTTP 429 or SSL error"}
  },
  "candidate_pool": {
    "raw": 18,
    "deduped": 12,
    "strong": 5,
    "weak": 4,
    "noise": 3,
    "sent_to_overlap_assessment": 9
  },
  "warnings": [
    "Semantic Scholar failed; results rely on PubMed/arXiv/OpenAlex.",
    "PubMed narrow query returned 0; broad query returned 8."
  ]
}
```

## 5. 检索策略

### 5.1 总原则

查新检索必须多路召回，不能只用一个检索式。

至少生成以下 5 类查询：

1. **宽主题查询**
   - 目标：找到同领域核心文献
   - 示例：`ischemic stroke CT segmentation`

2. **核心主题查询**
   - 目标：保留必要应用场景
   - 示例：`ischemic stroke non-contrast CT lesion segmentation`

3. **方法重合查询**
   - 目标：找方法是否已有
   - 示例：`medical image segmentation lightweight encoder domain generalization`

4. **上下文/约束查询**
   - 目标：找低资源、小样本、跨域泛化是否已有
   - 示例：`few-shot stroke CT segmentation domain generalization`

5. **baseline/竞品查询**
   - 目标：找与用户已知对比方法相关的直接工作
   - 示例：`nnU-Net ischemic stroke CT segmentation`

### 5.2 PubMed 策略

PubMed 不要把所有概念强 AND 到一起。建议：

- broad query：只 AND 核心医学对象、模态、任务
- method query：医学对象 + 任务 + 方法关键词
- 不要在 broad query 中强制 `lightweight`、`domain generalization`、`few-shot`

医学示例：

```text
Broad:
(ischemic stroke[Title/Abstract] OR cerebral infarction[Title/Abstract])
AND (CT[Title/Abstract] OR computed tomography[Title/Abstract] OR NCCT[Title/Abstract])
AND (segmentation[Title/Abstract] OR delineation[Title/Abstract])

Method:
(stroke[Title/Abstract] OR cerebral infarction[Title/Abstract])
AND (segmentation[Title/Abstract])
AND (deep learning[Title/Abstract] OR neural network[Title/Abstract])

Context:
(medical image segmentation[Title/Abstract])
AND (domain generalization[Title/Abstract] OR few-shot[Title/Abstract] OR low-resource[Title/Abstract])
```

### 5.3 arXiv 策略

arXiv 对医学临床主题噪声高，但对 AI 方法有价值。策略：

- 医学临床主题下，arXiv 默认作为辅助源
- 查询应更偏方法：`stroke CT segmentation deep learning`
- arXiv 返回结果必须经过轻筛；不要因 arXiv 噪声影响总体诊断

### 5.4 Semantic Scholar / OpenAlex / Crossref 策略

这些源适合 broad 和 method 查询。建议失败时不中断任务，但在 diagnostics 中展示错误。

如果 Semantic Scholar 频繁 429/SSL：

- 支持配置 `PAPER_SEARCH_MCP_SEMANTIC_SCHOLAR_API_KEY`
- 失败后自动降级到 OpenAlex/Crossref
- 前端报告显示“Semantic Scholar 不可用，本次结果覆盖不足”

## 6. 候选池策略

### 6.1 不再只使用 qualified/needs_review

查新应构建自己的候选池：

```text
strong_candidates:
  核心概念命中充分，来源稳定，有标题/摘要

weak_candidates:
  命中部分核心概念或方法概念，但缺少某些条件

source_noise:
  明显跑题，用于诊断，不进入 LLM 重合判断
```

发送给 `NoveltyCheckWorkflow` 的候选应为：

```text
strong_candidates + weak_candidates[:N]
```

默认不要把明显 source_noise 发给 LLM，以免污染判断。

### 6.2 保留 rejected 诊断

即使候选被过滤，也要保存：

- title
- source
- rejected reason
- matched query_id

报告里可显示“检索到 5 条 arXiv，但均为机器人/长上下文/生成模型等跑题文献，未进入查新对照”。

## 7. 查新判断策略

### 7.1 判断维度

不要只看关键词相似度。每篇候选应按维度比较：

```json
{
  "target_problem_overlap": "same|similar|different|unknown",
  "data_or_population_overlap": "same|similar|different|unknown",
  "method_overlap": "same|partial|different|unknown",
  "application_context_overlap": "same|similar|different|unknown",
  "baseline_or_evaluation_overlap": "same|partial|different|unknown",
  "claimed_contribution_overlap": "high|partial|adjacent|none|unknown"
}
```

### 7.2 结论规则

建议后端最终 risk_level 用规则聚合，不完全交给 LLM：

- high：同问题 + 同数据/场景 + 同核心方法/机制
- moderate：同问题 + 方法或约束部分重合
- low：同领域/相邻任务，但未见核心创新点重合
- unknown：候选太少、来源失败、摘要不足或检索覆盖不足

### 7.3 文案约束

禁止输出：

- “证明创新”
- “完全原创”
- “没有任何重合”

建议输出：

- “当前开放来源下未见明显重合”
- “需要扩展数据库/全文核验”
- “该结论依赖标题摘要，不能替代正式查新”

## 8. 后端实施步骤

### 阶段 A：低风险改造，1 天

目标：不用大改架构，先解决候选池为空和用户无感问题。

改动：

1. `web_app.py::_run_novelty_check_job`
   - 即使 `include_filtered_references=false`，也保留少量低风险 rejected 作为 diagnostics。
   - 如果 qualified + needs_review 为 0，但 raw_count > 0，则返回“检索到但全部过滤”的诊断。

2. `novelty_check.py`
   - `extract_claims()` 增加过滤规则，去除：
     - `验收重点`
     - `测试要求`
     - `报告应提示`
     - `建议来源`
     - `检索模式`

3. 前端报告
   - 显示 source_results 和 errors。
   - 显示 raw_count、kept_count、filtered_count。

验收：

- 之前截图中的任务不能只显示“候选总数 0”，必须显示：
  - arXiv 返回 5 条
  - Semantic Scholar 失败
  - PubMed 0 条
  - arXiv 5 条均被判为跑题

### 阶段 B：查新专用 Planner，2-3 天

新增：

```text
src/research_agent/novelty_planner.py
```

包含：

- `InnovationPlan`
- `NoveltySearchQuery`
- `build_novelty_plan_with_llm()`
- `build_novelty_plan_rules_fallback()`
- `sanitize_innovation_text()`

接口：

```python
def build_novelty_plan(innovation_text: str, search_mode: str) -> dict:
    ...
```

验收：

- 对同一个创新点生成至少 5 条查询。
- PubMed 至少包含 broad 和 method 两条。
- broad query 不强制所有创新元素同时出现。

### 阶段 C：查新专用 Search Runner，2-3 天

新增：

```text

```

包含：

- `run_novelty_search_plan(plan, sources, year)`
- 多 query 执行
- 合并去重
- query_id 追踪
- strong/weak/noise 分层
- diagnostics 汇总

复用：

- `paper_search.run_paper_search_backend_by_source`
- `paper_search.normalize_search_payload`
- `paper_search.dedupe_papers`
- `reference_screening.screen_reference`

不要复用：

- `_run_literature_search_pipeline()` 的最终 qualified/needs_review 作为唯一候选

验收：

- 同一测试样例至少能返回 broad_topic 下的真实相关候选。
- 即使某个源失败，也能完成任务并显示诊断。

### 阶段 D：重合判断升级，2 天

改造：

```text
src/research_agent/novelty_check.py
```

新增维度式比较输出：

```json
{
  "dimension_overlap": {
    "target_problem": "...",
    "data_or_population": "...",
    "method": "...",
    "application_context": "...",
    "evaluation": "..."
  }
}
```

LLM 只负责逐篇解释，最终 overall risk 由后端规则聚合。

验收：

- 对“卒中 NCCT 分割 + 轻量编码器 + 跨域一致性”样例，能区分：
  - 卒中 CT 分割文献：任务高重合
  - 通用医学图像轻量模型：方法部分重合
  - 机器人/arXiv 噪声：不进入主要对照

## 10. 验收标准

### 功能验收

- 输入中文创新点，系统能生成英文检索计划。
- 检索式不是单条，而是多组。
- PubMed 不因为过窄 AND 返回 0 后直接结束。
- 来源失败不导致任务失败。
- 候选为空时必须解释原因。
- 报告不声称“证明创新”。

### 质量验收

使用 `创新点查新测试集.md` 中 12 条样例：

- 01 医学影像 AI：应至少返回卒中/CT/分割相关候选，不能直接 0 对照。
- 03 法律 RAG：应返回 legal RAG / QA / hallucination 相关候选。
- 09 低重合压力测试：允许低重合或证据不足，但必须解释来源覆盖。
- 10 信息太少：应提示输入过泛，不能输出强判断。

### 工程验收

- `python -m pytest tests\test_core_smoke.py -q` 通过。
- `pnpm build` 通过。
- 历史记录中保存 plan、diagnostics、comparisons。
- `logs/search_audit...json` 或 novelty audit 能追踪每条 query 的返回数量。

## 11. 推荐优先级

优先做：

1. 阶段 A：诊断信息 + 输入清洗 + 空候选解释。
2. 阶段 B/C：查新专用 planner/search runner。
3. 阶段 D：维度式重合判断。

不建议优先做：

- 只调 prompt 让普通检索式更宽。
- 只默认勾选“评估已过滤候选”。
- 只增加每源数量。

这些只能缓解个别问题，不能解决查新任务和普通检索任务目标不同的根本矛盾。

## 12. 给接手开发者的一句话

查新不是“检索一个精确主题后总结”，而是“围绕一个创新主张构建多层证据网，再判断哪些已有工作与它在问题、方法、场景、数据和评价上发生重合”。后端要先保证候选池覆盖，再做重合判断；否则 LLM 再强也只能对空候选输出“证据不足”。
